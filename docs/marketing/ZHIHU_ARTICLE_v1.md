# 我做了一个给 AI Agent 用的 macOS OCR 工具：既能当 MCP，也能当 Skill

最近我把一个自己一直在用的小工具整理成了一个正式项目，叫 `ocrtool-mcp`。

它的核心能力很简单：

- 让 AI Agent 直接读取截图、收据、扫描件、图片里的文字
- 本地运行，不依赖云端 OCR
- 基于 macOS 原生 Vision Framework

但这个项目真正让我觉得“值得写出来”的地方，不只是 OCR 本身，而是我在做的过程中逐渐想明白了一件事：

**如果你想让一个能力真正被更多 Agent 和更多普通用户用起来，只做 MCP 其实不够，还需要考虑 skill 这一层。**

所以现在这个项目不是单纯的 “一个 MCP OCR server”，而是同时支持两种使用方式：

1. 作为 MCP server 接入支持 MCP 的客户端
2. 作为一个可安装的 skill，让支持 skills 的 Agent 直接使用

这篇文章我想讲的，就是这个项目是怎么来的，我为什么把它从“纯 MCP 工具”改造成了“Skill + MCP 双入口”，以及这件事对普通用户到底有什么意义。

---

## 一、我为什么会做这个东西

我自己的使用场景很直接：

- 截图里有一段文字，想让 Agent 继续分析
- 收据、票据、文档照片里有内容，想提取出来整理
- 某个界面不能复制文字，只能截图再交给 Agent
- 代码截图、终端截图、网页截图，希望直接转换成可处理文本

以前做这件事通常有几种路：

1. 手动 OCR，再复制粘贴给 AI
2. 用在线 OCR 服务
3. 用一堆 Python 工具和脚本自己拼
4. 调云端 API

这些方案不是不能用，而是都不够顺手。

我更想要的是：

- 在本地完成
- 尽量少依赖环境
- 能直接接到 Agent 工作流里
- 对用户来说不是“先学一套协议”，而是“装上就能用”

所以我做了 `ocrtool-mcp`。

---

## 二、它一开始其实只是一个 MCP 工具

项目最早的定位非常纯粹：

> 做一个实现了 Model Context Protocol 的 OCR server。

它用 Swift 写，跑在 macOS 上，直接调用 Vision Framework 做 OCR，然后通过 JSON-RPC / stdio 把能力暴露给支持 MCP 的客户端。

这样做的好处很明显：

- 对支持 MCP 的客户端很标准
- 很容易被 Claude Desktop、Cursor、Continue、Windsurf、Cline、Cherry Studio 这类工具接入
- Agent 可以把 OCR 当作一个标准工具来调用

从工程角度看，这其实是很干净的方案。

但问题也很快出现了：

**工程上干净，不等于用户上好用。**

---

## 三、MCP 对开发者很自然，对普通用户不一定

如果你平时就在接 MCP server，你会觉得这套逻辑非常合理：

- 给一个 command
- 配一个配置文件
- 把 tool 挂进 Agent
- 然后在对话里调用

但如果你从普通用户的角度看，真正的问题往往不是：

- “这个 server 有没有实现 JSON-RPC”

而是：

- “我到底该装什么？”
- “我这个 Agent 是支持 skill，还是支持 MCP？”
- “我想让它识别图片，是不是还要自己拼配置？”

也就是说，用户脑子里想的不是“协议”，而是“入口”。

这就是我后来决定改项目方向的原因：

**能力层可以是 MCP，但用户入口不能只剩 MCP。**

---

## 四、所以我把它改成了 Skill + MCP 双入口

现在这个项目有两层：

### 1. 底层是 OCR 核心 + MCP server

这一层负责真正的 OCR 能力：

- 输入可以是本地图片路径、URL、base64
- 输出可以是纯文本、Markdown 表格、结构化 JSON、代码注释
- 本地执行，走 macOS Vision

这部分依然保留为标准 MCP 能力，因为它适合做“工具层”。

### 2. 上层加了一个 `ocr-workflow` skill

这层是给支持 skills 的 Agent 用的。

它不是重新实现 OCR，而是把底层 `ocrtool-mcp` 二进制包装成一个更自然的工作流入口。

换句话说：

- **MCP** 解决的是 “Agent 如何调用工具”
- **Skill** 解决的是 “用户如何自然地使用这个能力”

这两者不是替代关系，而是互补关系。

---

## 五、为什么我觉得这才更接近真实世界的 Agent 产品

如果你真的想让一个能力被更多 Agent 使用，通常会遇到三类平台：

### 第一类：更擅长 MCP 的平台

这类平台天然就鼓励你接 server、暴露 tools。

对它们来说，`ocrtool-mcp` 作为 MCP server 就很合适。

### 第二类：更擅长 skill 的平台

这类平台更像“给 Agent 一套工作方法”。

它们可能并不要求用户理解 MCP，只需要：

- 安装一个 skill
- 然后直接说“帮我识别这张图片”

### 第三类：普通用户根本不关心协议

他们只关心一件事：

> 这个 Agent 到底能不能看图里的字。

这时 skill 往往比 MCP 更像“可消费产品”，而 MCP 更像“底层技术接口”。

我现在越来越觉得：

**如果一个项目最终想服务的是 Agent 生态，就应该同时考虑“协议兼容性”和“用户入口设计”。**

---

## 六、这个项目现在能做什么

目前 `ocrtool-mcp` 已经支持：

- macOS 原生 OCR
- 中英混合识别
- 本地路径 / URL / base64 三种输入
- 文本 / Markdown / JSON / 注释 四类输出
- 通用 MCP 接入
- skill 安装与工作流包装
- release 自动化和 Homebrew formula 同步

对于普通用户来说，最重要的其实是下面这个判断：

### 如果你的 Agent 支持 skills

优先装 skill。

因为这会让使用方式更接近自然语言工作流。

### 如果你的 Agent 支持 MCP server 配置

直接接 MCP。

因为这会让 OCR 作为标准工具暴露给 Agent。

### 如果两者都支持

我自己的建议是：

- 普通用户优先 skill
- 偏工程集成优先 MCP

---

## 七、普通用户到底该怎么装

这是我这次专门补到 README 里的部分，因为我发现很多项目都默认用户“知道自己在做什么”，但真实情况不是这样。

现在最简单的理解方式是：

### 路线 A：Skill 模式

适合支持 skills 的平台，例如某些 Codex / Claude 场景。

安装方式类似：

```bash
./scripts/install-skill.sh codex
```

或者：

```bash
./scripts/install-skill.sh claude
```

然后用户就可以直接说：

- “提取这张图片里的文字”
- “把这张截图 OCR 成 markdown 表格”
- “识别图片里的文本并转成 Python 注释”

### 路线 B：MCP 模式

适合支持 MCP 的平台，例如 Claude Desktop、Cursor 等。

本质上就是把 `ocrtool-mcp` 可执行文件接到 MCP 配置里，让 Agent 把 `ocr_extract_text` 当作工具调用。

---

## 八、我觉得这个项目最有意思的部分，不是 OCR，而是“包装方式”

单看 OCR，本身不是什么新东西。

Vision Framework 早就有，OCR 也不是新需求。

但我觉得这个项目有意思的点在于：

**它比较完整地体现了一个 Agent 能力从“可调用”走向“可使用”的过程。**

也就是：

- 先有底层能力
- 再有标准协议
- 再往上做用户可理解、可安装、可传播的入口

很多项目做到 MCP 就停了。

但如果你真的想让普通用户用起来，下一步一定会碰到：

- 怎么安装
- 怎么理解
- 怎么让用户不需要知道协议细节

而这正是 skill 层的价值。

---

## 九、接下来我还想继续做什么

目前这个项目已经能作为一个完整可发布的工具使用，但我后面还想继续完善几件事：

- 更好的批量 OCR 工作流
- 更多面向实际使用场景的 skill 封装
- 更明确的平台适配说明
- 更完整的“普通用户安装路径”设计

本质上我想验证的是：

> 一个 Agent 能力，怎样才能既保持工程上的干净，又对普通用户足够友好。

---

## 十、结语

如果你也在做 Agent 相关工具，我很推荐认真区分这两件事：

1. 你的能力是否“能被调用”
2. 你的能力是否“能被普通用户使用”

MCP 更偏前者。

Skill 更偏后者。

而真正好用的项目，往往需要同时考虑这两层。

`ocrtool-mcp` 现在就是我在这个方向上的一个小实验：

- 底层是本地 OCR
- 中间层是 MCP
- 用户层是 skill
